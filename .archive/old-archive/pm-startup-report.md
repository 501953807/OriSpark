## 一、项目背景与评估摘要

### 1.1 评估概况

2026-06-12，团队完成了对 OriStudio v3 系统的 9 角色独立评估：
- **6 位创作者**: 林山海(插画师)、赵一鸣(视频)、周巧手(手工)、陈摄影(摄影)、苏旋律(音乐)、文墨(文字)
- **1 位新手**: 小明(入门3个月)
- **1 位架构师**: 15年SaaS/创作工具经验
- **1 位律师**: 12年IP/数字合规经验

### 1.2 核心发现（一句总结）

**系统是为插画师设计的全链路工具，但在试图覆盖 6 种创作者时，每种都照顾到、每种都不深入。非插画师的 5 类创作者均给出了不及格评价。**

### 1.3 代码-文档匹配度（现场勘测结果）

| 维度 | 文档声称 | 代码实际 | 差异 |
|------|---------|---------|------|
| 数据模型 | 33 表 | **50 个模型类**(10个文件) | 文档少写 17 个 |
| API 端点 | 145 个 | **231 个** (230 REST + 1 WS) | 文档少写 86 个 |
| 路由文件 | 文档未提及完整清单 | **13 个路由器文件** | 文档未覆盖 |

**结论**: 代码比文档更完整，但这也意味着文档严重偏离实际代码，新成员无法通过文档了解系统真实规模。

## 二、结构化问题清单

### P0 — 致命级（不可上线，必须修正后才能发布）

#### P0-1: 系统架构聚焦偏差 — 只为插画师设计
- **评估来源**: 林山海、赵一鸣、周巧手、陈摄影、苏旋律、文墨（6位创作者全部）、架构师
- **影响范围**: 全部 7 个模块，核心是 [1]创意资产、[4]商业转化引擎、[5]内容分发中心
- **需要谁来解决**: 架构师(主)、业务专家(辅助)、程序员(确认代码边界)
- **具体表现**:
  - 赵一鸣(视频): "视频不是单一文件，是一个项目包。只接受 MP4/MOV 等于告诉插画师你的作品就是导出的 JPG"
  - 周巧手(手工): "系统是为'数字文件→印到产品上'设计的。手工艺人是'先做实物→拍照→照片上线'"
  - 苏旋律(音乐): "系统对音乐产业一无所知。PRO 注册/机械版税/ISRC 码全无"
  - 文墨(文字): "系统由视觉创作者为视觉创作者设计。文字作者是'贴上去的'"
  - 架构师: "最大架构错误 — 试图用一个统一数据模型服务 6 种根本不同的创作者"
- **解决方向建议**:
  1. v1 明确聚焦**插画师/AIGC 艺术家**单一类型，做到极致
  2. 其他5类创作者在 UI 上明确标记为"规划中（敬请期待）"
  3. 数据模型为未来扩展预留扩展点(creator_type 字段 + JSON 扩展字段)，但不实现专有工作流
  4. 实施计划从 6 天 6 阶段改为单创作者类型的深度实现

#### P0-2: Canvas 效果预览不及商业级
- **评估来源**: 林山海、架构师
- **影响范围**: [4]商业转化引擎 — 产品设计器 Step4
- **需要谁来解决**: 设计师(UI降级方案)、架构师(技术路线评估)、程序员(实现)
- **具体表现**:
  - 当前设计: 2D Canvas drawImage 叠加
  - 林山海: "T恤有褶皱/弧度/布料纹理，2D drawImage 做不了真实 mockup"
  - 架构师: "扁平叠加可行，照片级需要 CV。不能虚假承诺"
- **解决方向建议**:
  1. **选项A(推荐短期限)**: 接入 Printful Mockup Generator API (已提供真实产品照片级效果图)，用户选产品时从 Printful 拉取规格+效果图模板
  2. **选项B(中期)**: 预置10-20个高质量产品模板照片(真实拍摄的T恤/卫衣/马克杯等)，用 Canvas 做位移变换实现"设计稿贴在产品模板指定区域"
  3. **选项C(降级)**: 如果 A/B 都不可行，诚实降级为"产品规格预览表"——显示产品类型/颜色/尺寸/DPI匹配状态，不做视觉 mockup，引导用户去 POD 平台预览
  4. **必须写明技术能力边界**: 文档和 UI 中明确"预览为示意，以 POD 平台实际效果为准"

#### P0-3: POD 对接是"假对接"——URL 记录非 API
- **评估来源**: 林山海、周巧手、陈摄影
- **影响范围**: [4]商业转化引擎 — POD 发布子模块
- **需要谁来解决**: 业务专家(定义能力边界)、架构师(确认可对接API)、程序员(实现)
- **具体表现**:
  - 当前设计: "产品上架标记(URL 记录)"——仅存一个链接
  - 林山海: "没有 API 就不叫对接"
  - 周巧手: "Etsy 完全不管，但 50% 收入来自 Etsy"
  - 陈摄影: "图库集成纯纸面，发部流程写着'手动执行'"
- **解决方向建议**:
  1. **诚实重命名**: "POD对接" → "POD渠道管理(手动上架+链接追踪)"
  2. **v1 真实能力**:
     - Printful: 接入 Product Catalog API(获取品类/规格/定价) + 创建产品草稿
     - 印鸽: 如有开放API，同 Printful 模式；如无，手动记录链接
     - Redbubble/Printify: 手动记录链接
  3. **必须写入文档**: "当前版本不支持自动同步库存/订单/物流，不支持 API 级全自动上架"
  4. **诚实体现**: 在产品设计器 Step7"创建产品"环节明确标注"将引导您前往 Printful 完成上架"

#### P0-4: IP 登记模块的 UPL（无证法律服务）风险
- **评估来源**: 法律顾问
- **影响范围**: [3]IP 登记工作站 — 类别推荐引擎、智能助手
- **需要谁来解决**: 律师(审定合规方案)、业务专家(重写功能边界)、程序员(修改推荐逻辑)、设计师(UI 警告)
- **具体表现**:
  - "商标分类推荐 + 可注册性评估 + CNIPA 自动提交 = 自动商标代理"
  - "正是中国律师法禁止的'无证从事法律服务'"
  - 风险等级: **关键 × 可能性中 = P0**
- **解决方向建议**:
  1. **尼斯分类推荐改造**: 从单一推荐 → "根据您的描述，以下多个分类可能有相关性(附置信度百分比)"，多条替代方案展示，不可只推荐一条
  2. **CNIPA 提交流程**: 必须插入不可绕过的"建议咨询专业商标代理/律师"步骤，提供附近代理机构查询链接
  3. **所有 IP 登记指引**: 添加醒目免责声明(黄色警告框，不可折叠)，明确"本工具提供信息参考，不构成法律建议"
  4. **不能提供**: 可注册性评估(如"成功率85%")、单一分类推荐、自动提交、费用报价作为"报价"
  5. **7项必须补全的免责声明**:
     - 不构成律师-客户关系
     - 不构成法律建议
     - 不保证注册成功
     - POD 平台条款警告(当前完全缺失)
     - AI 内容标注要求
     - 侵权监测局限性
     - 司法管辖区限制

#### P0-5: 非视觉类型核心工作流完全缺失
- **评估来源**: 赵一鸣(视频)、苏旋律(音乐)、文墨(文字)
- **影响范围**: [1]创意资产中心、[4]商业转化引擎、[5]内容分发中心
- **需要谁来解决**: 业务专家(定义缺失工作流)、架构师(数据模型设计)、程序员(预留扩展点)
- **具体表现**:
  - 视频: 无工程文件支持(.prproj/.aep/.drp)、无视频指纹实现、无字幕管理、无审片协作、无横竖屏版本管理
  - 音乐: 无 ISRC 码管理、无 Split Sheets、无 Master vs Publishing 权利区分、无 Sample Clearance、无专辑/EP/Single 概念
  - 文字: 无章节结构(卷→章→节)、无 EPUB/PDF 排版导出、无章节级修订跟踪
- **解决方向建议**:
  1. v1 中**完全移除**音乐/视频/文字的专属功能承诺
  2. UI 中的创作者类型选择保留，但非插画类标记为"🎨 插画师/AIGC 艺术家(已完整支持) | 📹 视频(规划中) | 🎵 音乐(规划中) | ✍️ 文字(规划中) | 📷 摄影(规划中) | 🎭 手工(规划中)"
  3. 数据模型预留扩展点，但不实现专属工作流
  4. 修订所有设计文档，诚实标注每个功能的创作者类型适用范围

#### P0-6: 数据手动录入过量 — 用户体验灾难
- **评估来源**: 林山海、陈摄影
- **影响范围**: [1]创意资产中心(元数据)、[4]商业转化引擎(收入)、[6]经营管理中心(全模块)
- **需要谁来解决**: 程序员(自动抓取实现)、设计师(降低录入负担的UI)
- **具体表现**:
  - 林山海: "手动填 synopsis/completion_date/creation_tool — 让我做 Excel 文员"
  - 陈摄影: "1000 张照片逐一操作"
  - 当前设计: synopsis、completion_date、creation_tool 均为手动填
- **解决方向建议**:
  1. 自动元数据抓取: 文件名 → 标题(去除扩展名+格式化)，EXIF → creation_date/creation_tool/GPS，ffprobe → 视频参数，mutagen → 音频参数
  2. 批量导入: 支持文件夹递归读取，按文件夹名自动创建项目组
  3. 批量元数据模板: 预设作者/版权/许可证信息，批量应用到选中作品
  4. 收入: 支持 CSV 导入(支付宝/微信/POD平台导出格式)
  5. 不强制填写: 所有字段可为空，不阻塞流程

#### P1-3: 无入门引导与新手教程
- **评估来源**: 小明(新手)
- **影响范围**: 全局用户体验 — 首次使用体验
- **需要谁来解决**: 设计师(Onboarding 流程)、程序员(实现)
- **具体表现**:
  - 小明: "一打开全是空图表 → 感觉自己失败了还没开始"
  - "如果第一步是填法律表格 → 立即卸载"
  - "没有教程 → 立即卸载"
  - "9个专业术语只能理解3个"(SHA-256/EXIF/DPI/POD/C2PA/WIPO 等)
- **解决方向建议**:
  1. 首次启动按创作者类型引导: "我刚开始 / 已有作品库 / 专业创作者"
  2. 3 步快速上手: 拖入第一个作品 → 看到自动生成的哈希+缩略图 → 引导去变现 Tab 创建第一个产品
  3. 空状态设计: 每个 Tab 替代"空表格"为"✨ 拖入作品开始"或"🎨 选择作品创建第一个产品"的引导卡片
  4. 术语解释: hover tooltip 提供简单中文解释(SHA-256 → "作品数字指纹，用于证明创作时间")
  5. "神奇按钮"设计: 首页一个醒目按钮"🚀 快速开始"，一键导入→自动存证→引导创建产品

#### P1-7: 空仪表盘与财务软件感
- **评估来源**: 小明(新手)
- **影响范围**: 全局 Dashboard/[6]经营管理中心
- **需要谁来解决**: 设计师(空状态+视觉风格)

  4. 首页重构: 展示最近作品缩略图(最核心内容)而非数据看板

#### P1-8: 商业约稿流程完全缺失
- **评估来源**: 林山海

  3. 约稿收入 → 纳入 RevenueRecord 手动登记

#### P1-9: 权利管理粒度不足（文字/音乐创作者强烈反馈）
- **评估来源**: 文墨、苏旋律
- **影响范围**: [1]创意资产中心 — 权利信息面板
- **需要谁来解决**: 架构师(数据模型)、业务专家(权利类型定义)
- **具体表现**:

  - 苏旋律: "Master vs Publishing 权利区分——音乐行业最基础的法律区分，系统用单个 rights JSON 覆盖"
- **解决方向建议**:
  1. v1 插画师场景下，当前 7 个许可证选项 + 3 个 toggle 足够
  2. rights JSON 字段保持灵活性，为未来创作者类型预留扩展空间
  3. 文档中标注"当前权利模型主要面向视觉作品"

### P2 — 需要级（重要但不阻塞 v1 发布，可在后续版本迭代）

#### P2-1: 移动端响应式
- **评估来源**: 林山海、小明
- **影响范围**: 全部前端页面
- **需要谁来解决**: 设计师(响应式布局)、程序员(实现)
- **具体表现**: "没有手机端 — 核心缺失"
- **解决方向建议**:

## 四、关键决策点清单

以下是需要团队研讨并达成共识的关键决策点。每个决策点标注了必须参与的角色和推荐方案。

### KD-1: v1 聚焦策略
- **问题**: 是否将 v1 明确聚焦为"插画师/AIGC 艺术家专属工具"，其他 5 个创作者类型标记为"规划中"？
- **必须参与**: PM、架构师、业务专家
- **推荐方案**: **是，明确聚焦**。理由：
  - 6 位创作者全部指出系统偏向插画师
  - 架构师认为"最大架构错误是试图用一个统一模型服务6种"
  - 如果 v1 继续假装支持全类型，首发口碑将极差
  - Canva/Figma 都是从单一场景做深做透再扩展
- **风险**: 暂时放弃其他创作者市场。但错误的"全支持"比诚实的"单聚焦"伤害更大
- **决策 deadline**: 项目启动当日

### KD-2: POD 真实能力边界
- **问题**: POD 自动对接做到什么程度？Printful API 是接入产品目录+创建草稿，还是只做链接记录？
- **必须参与**: 业务专家、程序员
- **推荐方案**: **Printful 产品目录 API(读)+创建产品草稿(写)**。印鸽如有 API 同方案，Redbubble/Printify 手动链接。文档中诚实标注能力边界。
- **风险**: Printful API 如有 breaking change 需要维护。手动链接则失去"自动上架"的核心卖点。

### KD-3: Canvas 效果预览降级方案
- **问题**: 评估报告指出 2D Canvas 做不了真实 mockup。采用哪个方案？
- **必须参与**: 架构师、设计师
- **推荐方案**: 优先接入 **Printful Mockup Generator API**（返回 JPEG 产品效果图），不自己实现。如 API 接入有困难，使用预置产品模板照片+Canvas 位移变换。
- **降级方案**: 如果 API 和模板照片都不可行，降级为"产品规格预览表"（文本+参数展示），引导用户去 POD 平台实际预览。

### KD-4: IP 登记法律合规边界
- **问题**: 在当前法律评估结论下，IP 登记模块哪些功能必须在 v1 移除/改造？
- **必须参与**: 法律顾问(最终裁定)、架构师、业务专家
- **推荐方案**:
  - 必须改: 尼斯分类 → 多推荐+置信度（不可单一推荐）
  - 必须加: CNIPA 提交前 "建议咨询律师" 步骤（不可绕过）
  - 必须加: 7 项免责声明（首次启动确认+关键操作前提示）
  - 必须移: 可注册性评估、"成功率"类评分
- **风险**: 如不做，项目可能面临法律诉讼

### KD-5: 修订范围与优先级
- **问题**: P0/P1/P2 三个级别共 30+ 个问题，v1 发布前必须完成哪些？
- **必须参与**: PM、架构师
- **推荐方案**:
  - **v1 门槛(P0)**: 8 个 P0 问题必须全部解决
  - **v1 目标(P1)**: 9 个 P1 问题尽力解决，至少完成 P1-1(数据模型)、P1-4(文档匹配)、P1-5(批量导入)
  - **v1+ 规划(P2)**: 15 个 P2 问题纳入 backlog，v1.1/v2 迭代
- **风险**: 如试图在 v1 解决所有问题，开发周期将失控

### KD-6: 非插画师类型在 UI 上的处理
- **问题**: 导航/设置中的创作者类型选择，是隐藏非插画类型、显示但置灰、还是显示并标注"规划中"?
- **必须参与**: 设计师、PM
- **推荐方案**: **显示但标注"规划中"**。好处：透明诚实，让用户知道未来方向，同时避免流失(用户可能期望见到自己类型后愿意等待)。交互: hover 显示"我们正在为[摄影师]优化体验，敬请期待"。
- **替代方案**: 隐藏非插画类型。好处：减少困惑。风险：插画以外的用户进入后找不到自己的场景，认为产品不相关。

### KD-7: 实施计划重塑
- **问题**: 当前 6 天 6 阶段计划未考虑评估反馈。如何重塑？
- **必须参与**: PM、架构师、程序员
- **推荐方案**: 废弃当前实施计划，在完成所有设计文档修订后重新规划。新计划应:
  1. 先完成 P0 设计文档修订
  2. 以 v1 聚焦插画师的视角重新排期
  3. 每个 Phase 包含设计评审 gate
- **依赖**: KD-1 聚焦策略必须先确定

### KD-8: "空状态"与新手体验的投入程度
- **问题**: Onboarding + 空状态 + Tooltip + 视觉风格调整是一个完整 sprint 的工作量。v1 投入多少？
- **必须参与**: PM、设计师
- **推荐方案**: v1 做最小可用的 Onboarding(3步向导+空状态引导卡片+5个核心术语 Tooltip)。视觉风格做色彩+圆角+间距的轻量调整（不重做组件库）。完整的设计系统升级放到 v1.5。

## 六、沟通与协作约定

### 6.1 评审 Gate

每个 P0 问题的解决方案在设计阶段需要通过**团队评审 gate**才能进入实施:
- **Gate 1**: 聚焦策略 + 技术方案 (KD-1~4) — 全团队
- **Gate 2**: 设计文档修订完稿 — 架构师 + 业务专家双签
- **Gate 3**: 法律合规专项审核 — 法律顾问签批

### 6.2 设计文档命名与版本

- 修订后的设计文档采用 `v4` 版本号
- 文件命名: `docs/modules-v4/01-creative-assets.md` 等
- 保留 v3 文档作为历史参考（新增 README 说明 v4 与 v3 的差异）

### 6.3 文档与代码同步机制

- 每个 router 文件头部注释: `# 对应设计文档: docs/modules-v4/XX-xxx.md`
- 代码与文档不一致 → 以代码为准修正文档，或评估是否需要修改代码
- PR Review checklist 增加: "设计文档已同步更新" 检查项
